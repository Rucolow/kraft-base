## R1. 退勤ボタン（規模S・FEのみ）

### 現状と実害（v2で訂正）
`endActiveSession` の呼び出しは (a) 同一端末での交代時の置換、(b) 04:00境界の自動
クローズのみ。(b) は**常時起動端末（受付iPad）では04:00に発火**し、それ以外の端末では
次回起動時に発火する（RootBootstrap が boundary を依存に持つため。v1の「常に翌日の
起動時刻」は誤り）。いずれにせよ**明示的な退勤手段が無い**ため:
- 「終わった」はLINEで伝えるしかない（依頼の背景）。
- 勤怠が膨張する: 18時退勤でも最良で翌04:00まで計上（約10時間過大）。

### 設計
1. `lib/shiftOps.ts` に **`endShift(deviceId)`** を追加:
   - active session を取得し、その `staff_id` から名前を解決（**引数で staff を渡さない** —
     呼び出し側が「タップ中の別人」を渡す誤用をシグネチャで封じる）。
   - `ended_at = nowIso()` で UPDATE ＋ timeline_entry INSERT
     `{kind:'system', body:'退勤 {staffName}（終了！！）', author_id: <sessionのstaff_id>}`。
     「終了！！」は依頼の核心（モーリーが引き継ぎ画面で見る合図）なので timeline 本文に残す。
2. **`closeStaleSessions` の ended_at を `shiftBoundaryIso()` にキャップ**（数行）:
   押し忘れた日の膨張上限が「次回起動時刻」→「04:00」に縮む。R1の勤怠正確性
   主張を押し忘れ日にも効かせる実効策。
3. UI（ShiftGate、ヘッダーの名前タップ→/shift が既存導線）:
   - `activeSession` があるとき「現在のシフト: {name}」カード＋
     **「シフトを終了する（退勤）」**ボタンを表示（共有=roster画面上部、個人=digest画面上部）。
   - タップ → `window.confirm('{name} さんのシフトを終了しますか？')`（**名前入り**。
     共有端末では誰でも押せるが、これは startShift の置換と同じ信頼モデルとして許容）。
   - 実行後の表示（v2で再設計）: 揮発する「完了画面」は作らない。
     **「本日退勤済み」永続カード**を出す — 直近の ended session が今日のシフト日内なら
     ShiftGate 上部に「お疲れさまでした！終了！！（{time} 退勤: {name}）」カードを表示。
     必要なクエリは ShiftGate に既にある previous-ended クエリで賄える。
     これによりリロード/開き直しでも「退勤できたのか？」とならない（個人端末の
     基底状態が「開始確認」画面に戻る問題への対策）。
4. 二重タップガード（busy state）。
5. **副作用の明示**: 退勤するとその端末は RequireApp により /shift 止まりになり、
   **/checkin（ゲストのセルフチェックイン）も開けなくなる**。遅着ゲスト対応が残って
   いる場合は退勤しない運用とし、ボタン下に注記
   「※終了するとチェックイン画面も閉じます。遅着対応が残っている間は終了しないでください」。
6. **WorkTime.tsx の脚注文言を更新**: 「名前をタップしてから、次の人がタップする
   （または翌朝4時）まで」→「退勤ボタン、次の交代、または翌朝4時まで」。

### 受け入れ条件
- demo e2e: **スタッフ（モーリー=staff）**でシフト開始→/shift→退勤→
  「本日退勤済み」カード表示→リロード後もカード残存→再度開始できる。
  /handover に「退勤 …（終了！！）」行。
  （owner_shift.cjs の拡張だけでは不可 — WorkTime は role='staff' のみ集計するため、
  勤怠の検証はスタッフ役で行うこと）
- WorkTime: 退勤した日の分数が start→end 実時間。同日の退勤→再出勤は2行に分かれ
  合算される（既存挙動、確認済み）。
- 押し忘れ日: 翌04:00でクローズされる（キャップの単体テスト: shiftOps or date のテストに追加）。

## R2. チェックイン時間「未定」（規模S・FEのみ）

### 現状
`guest.checkin_time` は text・null可。全使用箇所が文字列扱い（一覧 `IN {time・bed}`、
詳細 raw、ソートは `COALESCE(checkin_time,'~~')` の文字列ソート、CheckIn.tsx は不使用、
serialize 対象外 — 全て裏取り済み）。

**既存バグ（レビューで発見・R2と同時に直す）**: `<input type="time">` は非時刻テキストを
空にサニタイズするため、**checkin_time に自由テキストが入っているゲスト**（例: seed の
'遅着 ~19:30'）**を別件で編集して保存すると、時刻情報が黙って消える**。

### 設計（サンチネル方式・DB変更なし・v2で state 分離を明記）
1. GuestEdit に**「未定」トグルボタン**（BEDSボタンと同トーン）。
   - **未定は `form.checkin_time` とは別の boolean state で持つ**。time input の value に
     '未定' を決して渡さない（type=time のサニタイズと乖離するため）。
   - 保存時: `undecided ? '未定' : (form.checkin_time || 温存値 || null)`。
   - ロード時: `checkin_time === '未定'` → トグルON・input空。
2. **非時刻テキストの温存ガード**（既存バグ修正）: ロード値が HH:MM でも '未定' でも
   空でもない場合、input を触っていない限り既存値をそのまま保存する
   （同ファイルの `bentoTouched` と同型のパターン）。
3. 表示: 一覧・詳細は既存ロジックで自然に動くが、文字列連結では埋没するため
   **`<Badge>IN未定</Badge>` を GuestCard に追加**（貸切Badgeと同パターン・数行）。
   未定ゲストは「いつ来るか分からない＝一日中待ち受け」なので視認性が本質。
4. ソート: '未定' は末尾（時刻→未入力→未定）。1日1〜3組の宿では並びの実害は無く、
   視認性はバッジで担保するためこのままとする。

### 採らなかった案（v2で理由を強化）
- **datalist に「未定」を足す**: `type="time"` input は非時刻値を受け付けないため実装不能。
- **boolean列追加**: migration＋オーナーSQLが表示ニュアンスに対して過大。
- **null をそのまま「未定」と表示**: 過去分含む全 null 行（大半は「OTAからまだ転記して
  いない」）を遡って「未定」と誤ラベルしてしまう。

### 受け入れ条件
- e2e（roundtrip.cjs 拡張）: 未定ONで作成→一覧に「IN未定」バッジ→編集でトグルON復元
  →OFF＋時刻入力→通常動作。**自由テキスト温存**: '遅着 ~19:30' のゲスト（seed に実在）の
  弁当数だけ変えて保存→checkin_time が消えていない。

## R3a. ゲストの月カレンダー（規模M・FEのみ・**7月末までに届ける**）

1. **`useGuestsInMonth(ym)` を data/queries.ts に追加**（`stay_date BETWEEN 月初 AND 月末`）。
   既存クエリは「今日」と「明日以降」しか取れず、**今月の過去日すらカレンダーに出せない**
   ため必須工程（同期は global bucket 全件済みなのでクエリ追加のみ）。
2. /guests に第3タブ**「カレンダー」**（今日/これから先/カレンダー）。AppShell の
   6タブ上限とは無関係（ルート内タブ）。
3. 月グリッド（7列）。セル表示は **Q2の回答で確定**（組数 or 人数合計。実装コストは同一）。
   **貸切（whole_house=1）のゲストがいる日はセルを wood トーンで強調**（運用上の特異日）。
   キャンセルは既存 `isActive` で除外。
4. セルタップ → **グリッド下部に選択日のゲストリストを表示**（GuestCard 流用、
   選択日 state 1個）。「これから先リストへスクロール」案は廃止（今日以前の日付で
   スクロール先が存在しないため）。
5. 月ナビ（前月/翌月、初期=今月）。`addMonth`/`monthLabel` は WorkTime.tsx から
   `lib/month.ts` へ抽出して共用（この2関数限定。ユーティリティ集に育てない）。
6. 過去月も 1 のクエリで表示可能。

### 受け入れ条件
- demo e2e 新設: カレンダータブ→シードゲストの日にバッジ→タップで下部に名前表示。
  貸切日の強調。既存タブ回帰なし。**WorkTime の月ナビ回帰**（抽出リファクタの影響先）。

## R3b. スタッフのシフト予定管理（**まず R3b-lite で試す**・本実装は条件付き）

### R3b-lite（規模S・FEのみ・インフラゼロ）— 推奨の第一歩
`content` テーブル（INSERT/UPDATE = org member 済み）に `kind` 追加はせず slug 固定の
1行（例: slug='shift-plan-memo'）を置き、**R3a カレンダー画面の下部に自由記述の
「シフトメモ」欄**として表示・編集できるようにする。
「8/1 モーリー / 8/2-8/5 ルッコロー」のような手書き運用で「カレンダーの近くで
シフト予定が見える」を即日満たす。**Q3の回答が「まず試す」ならこちらを R3a と同PRで**。
※content.kind には check 制約があるため既存 kind（'manual' 等）を使うか、制約変更が
要るなら lite でも 0018 が必要になる — 実装時に 0007 の check 制約を確認し、
制約変更が必要なら「slug固定＋既存kind」で回避する。

### R3b-full（規模L・DB＋インフラ・lite の使用実績と🔶回答後）
1. **新テーブル** `shift_plan`: `id / date text / staff_id FK / note / created_by / created_at`。
   **date の意味は「シフト日（04:00境界）」と定義**（実績 shift_session と同じ帰属に
   しないと深夜勤務の突合が狂う）。
2. migration は **0010_product.sql をテンプレ**に: publication 追加＋
   **`grant select ... to powersync_role`**（漏らすと初期スナップショットが黙って失敗）。
3. **リポの `apps/staff/powersync/sync-rules.yaml`（正本）を先に更新**→その内容を
   PowerSync ダッシュボードへ貼って Deploy（re-snapshot リスク→営業時間外、
   L5使い捨てリハ or L3観測）。client schema.ts / staging/schema.sql /
   **engineering-principles §4 RLS↔UI表** も追随（§10 の二重管理ポイント）。
4. UI: R3a のセルにスタッフ名チップ、セルタップシートで割当。
5. serialize: text/uuid のみ→変換マップ変更なし（boolean/配列を足さないこと）。

**🔶 R3b-full 前の判断事項（v2で拡充）**:
- 編集権限（owner only / 全員）
- 日単位か**期間**（「8/2〜8/9 滞在」型ならスキーマが変わる）か。時間帯ラベルの要否
- **予定と実績（WorkTime）を同じカレンダーで突合したいか**
- 週単位の繰り返し入力の要否（無いと毎月手打ちで使われなくなる典型）
- 過去分の保持要否／通知連携の要否
