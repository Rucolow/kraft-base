## 0c. オーナー回答（2026-07-20）と確定事項

- **Q1（退勤の目的）**: 「システムで退勤管理をしたい。スタッフが手動でLINEを送ってくるのを
  **やめさせたい**。iPadで退勤を押したら『お疲れ様でした』が出るのがよい」
  → **通知連携は不要と確定**。アプリ内記録（timeline/勤怠）＋端末上のねぎらい表示で完結。
  モーリー自身はアプリ（引き継ぎ/勤怠）を見る側に回る。v2で懸念した「LINE代替の不全」は
  目的の取り違えで、実際は逆（LINE習慣の廃止が目的）。R1設計はそのまま成立。
- **Q2（セル表示）**: 「可能なら組数と人数の両方。窮屈なら人数合計」
  → **「N組M名」のコンパクト表記を第一候補**（例: `2組6名`）。430px幅で崩れる場合は
  人数のみ（`6名`）にフォールバック。md以上は常に両方。
- **Q3（シフト管理）**: 「**本格版**。オーナーの管理画面で、カレンダー表示で
  **シフトビューとゲストビューを切り替え**られると最高」
  → R3b-lite（自由記述メモ案）は**廃止**。R3b-full 一本に確定。
  カレンダーは1画面で「ゲスト/シフト」のビュー切替を持つ設計へ（§R3b）。

**v3で確定させた残り設計判断**（再質問せず既定値で確定。変更希望があれば実装前に一言）:

| 論点 | 決定 | 根拠 |
|---|---|---|

## R1. 退勤ボタン（規模S・FEのみ）

### 現状と実害
`endActiveSession` の呼び出しは (a) 同一端末での交代時の置換、(b) 04:00境界の自動クローズ
のみ（(b) は常時起動端末では04:00、それ以外は次回起動時に発火）。明示的な退勤手段が無く:
- 「終わった」の合図がLINE頼み（モーリーはこの習慣を廃止したい）。
- 勤怠が膨張: 18時退勤でも最良で翌04:00まで計上。

### 設計
1. `lib/shiftOps.ts` に **`endShift(deviceId)`**:
   - active session を取得し、その `staff_id` から名前を解決（引数で staff を渡さない —
     「タップ中の別人」を渡す誤用をシグネチャで封じる）。
   - `ended_at = nowIso()` UPDATE ＋ timeline_entry INSERT
     `{kind:'system', body:'退勤 {staffName}（終了！！）', author_id: <sessionのstaff_id>}`。
     「終了！！」は依頼の表現をそのまま帳簿に残す（モーリーが引き継ぎ画面で見る合図）。
2. **`closeStaleSessions` の ended_at を `shiftBoundaryIso()` にキャップ**（数行）:
   押し忘れ日の膨張上限を「次回起動時刻」→「04:00」に縮める。
3. UI（ShiftGate）:
   - `activeSession` があるとき「現在のシフト: {name}」カード＋
     **「シフトを終了する（退勤）」**ボタン（共有=roster上部、個人=digest上部）。
   - タップ → `window.confirm('{name} さんのシフトを終了しますか？')`（名前入り。
     共有端末では誰でも押せるが startShift の置換と同じ信頼モデルとして許容）。
   - 実行後: **「本日退勤済み」永続カード** — 直近の ended session が今日のシフト日内なら
     ShiftGate 上部に**大きめのねぎらい表示**「お疲れさまでした！終了！！（{time} 退勤: {name}）」。
     Q1の「押したらお疲れ様でしたが出る」をこのカードが担う。揮発する完了画面は作らない
     （リロード/開き直しでも残る＝個人端末の基底状態が開始確認画面に戻る問題への対策）。
4. 二重タップガード（busy state）。
5. **副作用の明示**: 退勤するとその端末は /shift 止まりになり **/checkin も開けなくなる**。
   ボタン下に注記「※終了するとチェックイン画面も閉じます。遅着対応が残っている間は
   終了しないでください」。
6. **WorkTime.tsx の脚注更新**: 「退勤ボタン、次の交代、または翌朝4時まで」。

### 受け入れ条件
- demo e2e: **スタッフ役（モーリー=staff）**で開始→退勤→退勤済みカード→リロード後も残存
  →再開始可。/handover に「退勤 …（終了！！）」行。
  （WorkTime は role='staff' のみ集計のため、オーナー役では勤怠を検証できない）
- WorkTime: 退勤日の分数が実時間。同日の退勤→再出勤は2行で合算（既存挙動）。
- 押し忘れ: 翌04:00でクローズ（キャップの単体テスト追加）。

## R2. チェックイン時間「未定」（規模S・FEのみ）

### 現状
`guest.checkin_time` は text・null可。全使用箇所が文字列扱い（裏取り済み: 一覧連結・詳細raw・
`COALESCE(...,'~~')` ソート・CheckIn不使用・serialize対象外）。
**既存バグ（同時修正）**: `<input type="time">` が非時刻テキストを空にサニタイズするため、
自由テキスト時刻（seed の '遅着 ~19:30' 等）を持つゲストを別件編集で保存すると**時刻情報が黙って消える**。

### 設計（サンチネル方式・DB変更なし）
1. GuestEdit に**「未定」トグル**（BEDSボタンと同トーン）。
   - **未定は `form.checkin_time` と別の boolean state** で持ち、time input の value に
     '未定' を渡さない（type=time のサニタイズと乖離するため）。
   - 保存: `undecided ? '未定' : (form.checkin_time || 温存値 || null)`。
   - ロード: `checkin_time === '未定'` → トグルON・input空。
2. **非時刻テキストの温存ガード**: ロード値が HH:MM/'未定'/空 のいずれでもない場合、
   input 未タッチなら既存値を保存（`bentoTouched` と同型）。
3. 表示: **`<Badge>IN未定</Badge>`** を GuestCard に追加（貸切Badgeと同パターン）。
   未定＝一日中待ち受けの例外なので、文字列連結での埋没を防ぐ。
4. ソート: '未定' は末尾（時刻→未入力→未定）。視認性はバッジで担保。

### 採らなかった案
datalist追加（type=time のため実装不能）／boolean列追加（表示ニュアンスに migration は過大）
／null=未定表示（未転記の既存全行を遡って誤ラベル）。

### 受け入れ条件
- e2e（roundtrip拡張）: 未定ONで作成→「IN未定」バッジ→編集でON復元→OFF＋時刻→通常動作。
  **温存**: '遅着 ~19:30' ゲストの弁当数のみ変更→checkin_time 不変。

## R3b. シフトビュー（規模L・DB＋インフラ・PR-3）

### データ・サーバー
1. **migration 0018**（**0010_product.sql をテンプレ**に、以下を1本で）:
   - `create table shift_plan (id uuid PK default gen_random_uuid(), date text not null,
     staff_id uuid not null references staff(id), label text, created_by uuid references staff(id),
     created_at timestamptz not null default now())`＋ `(date)` index。
     **date は「シフト日（04:00境界）」を指す**（実績と同じ帰属。コメントで明記）。
   - RLS: SELECT = `is_org_member()` / INSERT・UPDATE・DELETE = `is_owner()`。
   - `alter publication powersync add table shift_plan` ＋
     **`grant select on shift_plan to powersync_role`**（漏らすと初期スナップショットが黙って失敗）。
2. **リポの `apps/staff/powersync/sync-rules.yaml`（正本）に
   `- SELECT * FROM shift_plan` を追加**→その内容をPowerSyncダッシュボードへ貼って
   Deploy（re-snapshot リスク→**営業時間外**、L3観測: デプロイ後に拒否バッジ/同期を確認）。
3. クライアント: `schema.ts` に shift_plan（全列 text）追加／`staging/schema.sql` 追記／
   **engineering-principles §4 RLS↔UI表に行を追加**（書込=owner・UIゲート=isOwner）／
   serialize はマップ変更なし（**boolean/配列を足さないこと**）。
4. **デプロイ順序（重要）**: ① オーナーが 0018 SQL 実行 → ② sync-rules Deploy →
   ③ コードPRマージ。逆順だと書き込みが RLS/未知テーブルで拒否バッジ行きになる。

### UI（R3aの骨格に載せる）
5. ビュー切替「シフト」を解禁。シフトビューのセル: その日の `shift_plan` 行を
   **スタッフ名チップ**で表示（`staff.accent` で色分け、`label` があれば小さく併記）。
6. **編集（isOwner のみ・RLSと同一ゲート）**: セルタップ → 下部シートに
   - その日の割当一覧（削除可）
   - スタッフ選択＋label入力で追加
   - **「期間でまとめて入力」**（開始日/終了日＋スタッフ → N行一括INSERT。
     writeTransaction でまとめる — 複数行初期化の既定則）
   - **「前週をコピー」**（前週7日分の行を今週へ複製）
7. スタッフ（非オーナー）: シフトビューは閲覧のみ（編集UIを出さない — UIゲートは
   RLSと寸分違わず一致させる）。

### 受け入れ条件
- demo e2e: devSeed に shift_plan のシード（数行）を追加した上で —
  シフトビューにチップ表示／オーナーで追加・削除・期間一括・前週コピー／

- L3: 本番デプロイ後、オーナー端末でシフト1件追加→別端末に反映→拒否バッジが出ない。

## R4. 宿泊者数の表示修正（規模S・FEのみ・計画 2026-07-20）

### 依頼原文

> 代表者が１人で複数人の予約が入った時、スタッフツールでは宿泊者数じゃなく代表者数が
> 表示される。宿泊者数をパッと見てわかるようにしたい。

### 現状（コードで裏取り済み）

データは「1予約=1行（代表者名＋party_size）」。**行数（組数）を「名」と表示している
箇所が3つ**残っている。カレンダー（R3a）は既に正しく人数集計（N組M名）しているので、
リスト系をカレンダーの流儀に揃える修正。

| 箇所 | 現状 | 修正後 |
|---|---|---|
| Guests.tsx:78「本日のゲスト — N名」 | N=行数 | **「— N組M名」**（M=Σparty_size） |
| Guests.tsx:97「これから先」日付見出し「— N名」 | N=行数 | **「— N組M名」** |
| Today.tsx:107「本日のチェックイン X / Y」 | 行数ベース | **人数ベース「X名 / Y名」**（X=到着済み行のΣparty_size、Y=activeのΣ） |
| Guests.tsx タブ「今日 N」「これから先 N」 | 行数・ラベル無し | **人数M**に統一（見出しのMと一致させる） |

### 設計

1. `components/GuestCard.tsx` の `isActive` の隣に **`headcount(guests)`** を追加:
   `guests.filter(isActive).reduce((s,g)=>s+(g.party_size??1),0)`。
   集計の正本を1箇所にする（§10 二重管理の予防）。GuestCalendar の同型ロジックも
   これに置換（挙動不変のリファクタ）。
2. 上表の4箇所を headcount ベースに変更。キャンセル除外は従来どおり isActive。
3. 触らない: GuestCard の各予約「3名」表示／ゲスト詳細／カレンダー（既に正）／
   CheckIn の partySize（名簿人数、正）。

### 検証

- 既存 e2e との整合: sim_verify の「cancel lowers 今日 count」（1名の予約をキャンセル
  →数が1減る）と「header count matches tab」は、**両方を人数化すれば自然に通る**
  （タブと見出しのMが一致し、1名キャンセルで両方1減）。
- 追加 check（roundtrip.cjs が既に3名予約 Ronaldo を作るのでそこに追記）:
  作成後 /guests の「本日のゲスト」が **+3名**（かつ +1組）になっていること。
  Today のバッジが人数表記（「/ 」の右が Σparty_size）であること。
- 通常ゲート＋CI e2e 12スイート緑維持。実装時に標準の敵対レビューを1周。

### 受け入れ基準（モーリーの言葉で)

「3名1組の予約が入った日に、ゲスト一覧の見出し・タブ・ホームのチェックインバッジの
どれを見ても『3名』とパッと分かる（1名/1組とは表示されない）」

## R5. 「これから先」タブでの弁当表示（規模S・FEのみ・計画/実装 2026-07-23）

### 依頼原文（2026-07-23 モーリー）

> 今ゲストのカテゴリーから「今日」の部分だけお弁当の表示がわかるのですが、「これから」の
> ところで一覧でお弁当注文済みか見えたりしますか？

### 現状（コードで裏取り済み）

- `BentoDayPanel` は「今日」タブ（全機能版）とカレンダー選択日（compact版）にのみ配置。
  「これから先」タブは日付ごとのゲスト一覧のみで、弁当情報がゼロ。
- 未来分の注文データ自体は端末に同期済み（koguchi は前日締切前に受注 → 未来日付の
  bento_order 行が常に存在）。表示していないだけ。

### 設計

1. **単一レンジクエリ**: `useBentoOrdersAfter(shiftDate())` を親（`Guests`）で1本だけ張る。
   日付ごとに `useBentoOrdersForDate` を張ると watch が日数分増える（iPad で重い）ため、
   1クエリで全未来注文を取り、JS でグルーピングして各行に配る。照合ピッカー用の
   `useGuestsAroundDate` は「これから先」では張らない（＝この画面は読み取り専用）。
2. **日付ごとサマリー行**: `BentoSummaryLine`（compact 描画を純関数コンポーネントに切り出し）
   を各日付見出しの下に表示。`🍱 弁当注文 計N食（焼肉2・ベジタリアン1）未照合N件`。
   `now` は親の1タイマーから配る。
3. **ゲストカードの 🍱 チップ**: 照合済み注文を持つゲストのカードに `🍱 焼肉×2` を表示
   （モーリーの質問「誰が注文済みか一覧で」への直接の答え）。`GuestCard`/`GuestList` に
   オプショナル prop を足すだけ（今日タブ・カレンダーは無影響）。
4. **注文だけある日の取りこぼし防止**: 表示日付を「ゲスト日付 ∪ 可視注文のある日付」の
   和集合で構成。宿泊者ゼロで弁当だけの日（電話注文等）も見出し＋サマリーで surface。

### 採らなかった案

- 「これから先」での照合UI（フル `BentoDayPanel`）: ①watch が日数×2本に増える
  ②照合は「今日」タブ（実行可能ウィンドウ）に集約する既存設計、カレンダー compact も
  読み取り専用 → 一貫性のため read-only を維持。未来分の事前照合が必要になれば別PRで検討。

### 検証

- vitest: `bento.ts` 純関数は既存12テストでカバー（`guestOrderChip`/`totalMeals`/
  `mealsBySlug`/`isUnmatched`/`isVisibleOrder`）。ロジック追加なし。
- e2e（`bento.cjs`）: 「これから先」タブに切替え、①日付サマリー `計3食` ②`未照合1件`
  ③紐づけゲストの `🍱 焼肉×2` チップ ④注文だけの日（`おむすび3`）を検証。
  デモシードに未来ゲスト Sofia Lombardi（+2日）＋未来注文3件を追加。
